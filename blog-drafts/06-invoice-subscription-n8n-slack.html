不用自己盯到期日、不用手動追帳：n8n + Claude Code 把發票與訂閱費用串在一起

前言

這是 Stacktools 開發日記的第六篇。前幾篇我們談了內容工具、推薦文生成、IG 監控，這篇要轉換跑道——說說財務這件事。

說「財務工具」聽起來很嚴肅，實際上這篇要解決的問題很具體：發票要記得開、帳要記得追、訂閱費用要記得續。這三件事都不難，但全部要靠「記得」就很容易漏掉。這篇會介紹我怎麼用 Claude Code 做出這兩個工具，再用 n8n 把提醒自動推上 Slack，讓這些事不再需要靠記憶。

目標讀者跟之前一樣：正在學 Claude Code 或 n8n、對自動化有興趣，但不確定自己能做到這個程度的人。這篇除了介紹功能，也會說明是怎麼做出來的，讓你判斷自己能不能複製這條路。

問題背景

我們公司同時有兩個財務問題。第一個是發票開立與應收帳款追蹤。每次幫客戶完成一個專案，就要開一張電子發票，然後記錄「這張發票付了沒」。聽起來簡單，但如果同時有五六個客戶、每個客戶到期日不同，再加上有些客戶付款不準時，很容易就搞不清楚哪些收到了、哪些已經逾期還沒催。以前用 Google 試算表記錄，每次都要手動更新狀態，還要記得去查每張發票的到期日。

第二個問題是付費訂閱費用失控。我們同時訂了十幾個 AI 工具、開發工具、設計工具，加起來每個月花多少其實沒人說得清楚。有些工具是美金月付，有些是日圓年付，有些是台幣一次性購買——分散在不同幣別、不同週期、不同信用卡，真的沒辦法在腦子裡加總。更麻煩的是，技術部和行銷部各自訂了一些工具，費用責任不清楚，到底誰在用什麼、用多少，沒有一個地方可以集中看。

整體架構：哪些工具做哪些事

Claude Code 負責做這兩個工具的使用介面和資料邏輯。我把需求說清楚：「我需要一個可以開立電子發票、追蹤付款狀態的工具，支援草稿、待付、已付、逾期、作廢五種狀態」，然後再描述訂閱工具的欄位需求——包括幣別、計費週期、部門分類、自動續約開關。Claude Code 把這些畫面和資料處理全部做出來，我沒有自己寫任何一行程式碼。

發票開立這部分，工具需要真的能申請電子發票，不是只是記錄一筆資料。這裡串接了光貿電子發票 API——這是台灣常見的電子發票服務商，支援 B2B（打統編）和 B2C（無統編）開立。我描述了開立流程的邏輯，Claude Code 把 API 串接、簽名計算、金額拆稅全部處理好。點一個按鈕，就能真的送出去申請發票號碼。這件事如果自己查 API 文件來寫，大概要花大半天；我只需要把 API 說明貼給 Claude Code，讓它去理解格式並實作。

n8n 在這個架構裡負責定時做兩件事：一是每月跑一次，把所有有效訂閱彙整成一份雙週報告，分技術部和行銷部列表，算出本月預估費用，推到 Slack；二是每天早上檢查一次，如果有訂閱剩三天就到期，就發一條提醒訊息。這些都是用 n8n 的排程觸發節點加上一個 HTTP Request 節點去拿訂閱資料，再用 Code 節點組裝訊息、最後推到 Slack 頻道。整個流程我自己在 n8n 畫出來，Code 節點裡的邏輯交給 Claude Code 幫我寫。

Slack 是訊息的出口。報告和提醒統一推到指定頻道，不需要主動去查工具，有事情它就會出現。這個選擇很實際：我們本來就在用 Slack，不需要再多一個地方看通知。

發票工具：從草稿到確認收款

發票工具的操作流程分四個步驟。首先建立草稿，填寫客戶名稱、統一編號、品項明細、折扣，系統自動計算含稅金額（稅率 5%）和預設十天後的到期日。草稿狀態下可以隨時編輯。

確認資料無誤之後，點「開立發票」按鈕，工具會去呼叫光貿 API，把發票號碼拿回來存起來。這個過程對使用者來說只是點一個按鈕，背後的 API 簽名、金額計算、B2B 與 B2C 差異都是自動處理的。發票開立成功後，狀態從草稿變為待付款，並顯示發票號碼。

客戶匯款之後，點「確認收款」，填入實際匯款日期和匯款帳號末五碼，狀態就更新為已付款。這筆帳號末五碼是日後對帳的依據，不是純粹的備註欄位。如果發票開立之後出了問題，也有「作廢」功能，同樣會透過 API 送出作廢申請。

頁面上方有三個統計卡片，隨時顯示目前有多少筆待付款、多少筆逾期未付、本月已收款多少金額。逾期狀態是自動同步的——每次開啟頁面，工具會檢查所有待付發票的到期日，超過今天的自動標記為逾期。

訂閱監控：分部門、算月費、追到期日

訂閱監控工具讓你集中管理所有付費訂閱。每筆訂閱可以設定服務名稱、分類（AI 工具、開發工具、設計工具、雲端儲存、其他雜支）、費用金額、幣別（台幣、美金、歐元、日圓）、計費週期（月付、年付、一次性）、下次續約日，以及是否自動續約。

介面上的核心設計是按部門分區：技術部和行銷部各自顯示在一個卡片裡，每個卡片的標題會即時顯示「這個部門目前有幾項有效訂閱、本月費用多少」。這個設計來自實際需求——兩個部門的工具費用本來就應該分開看，不是混在同一張表裡。

費用統計會自動換算成台幣。年付的訂閱會除以 12 換算成月費估算，讓你能一眼看出真實的月度負擔。頁面頂端的四個統計卡片顯示本月費用估算、年度費用估算、有效訂閱項目數，以及七天內即將到期的項目數——如果有即將到期的，那個卡片會顯示橘色並輕微閃動，視覺提示你要注意。

每筆訂閱還可以記錄登入帳號、密碼、支付者是誰。密碼欄位預設隱藏，點「顯示」才看得到，這樣截圖或分享畫面的時候不會直接外露。自動續約的開關可以在列表頁直接切換，不用點進去編輯。

n8n 串 Slack：兩種自動通知

光是把資料存下來還不夠，重點是「需要注意的時候你能收到通知」。這裡用 n8n 串了兩條流程，都推到同一個 Slack 頻道。

每月雙週報告

第一條流程是定時觸發，每個月固定在指定日期早上九點跑一次。流程很直接：觸發之後去呼叫訂閱工具的 API 拿資料，然後用一個 Code 節點把資料組裝成報告格式，最後送到 Slack。

報告的格式經過幾次調整。一開始只是一條條列點，後來改成有框線的表格，技術部和行銷部各一張，欄位包括服務名稱、分類、金額、週期、月費換算。Slack 本身不支援一般的表格語法，但可以用等寬字體的程式碼區塊來模擬有框線的表格效果——這件事我交給 Claude Code 處理，它把每一欄的顯示寬度計算（包含中文字元佔兩格的問題）都考慮進去了。

報告最上面是費用摘要一行：本月預估、年度預估、有效訂閱數。如果當月有訂閱七天內到期，還會出現一個🔴的區段特別標出來，手動續約的訂閱會加上 ⚠️ 標記。

每日三天到期提醒

第二條流程是每天早上九點跑，但只有在有訂閱剛好剩三天到期的時候才發訊息。判斷邏輯：拿到所有有效訂閱之後，篩出下次續約日距離今天剛好是三天的項目，如果一個都沒有，流程靜默結束，什麼都不發。這樣不會每天轟炸 Slack，只在真的需要的時候出現。

訊息格式很精簡：哪個服務、費用多少、哪天到期。如果這個訂閱的自動續約是關閉的，訊息裡會加上「需手動續約」提示，讓人一看就知道要主動去處理，不能等它自己扣款。

兩條流程在 n8n 裡是平行的——共用同一個 workflow，但由兩個獨立的觸發節點驅動，互不干擾。

一個讓我重新想清楚的時刻

這個工具做到一半，我把 n8n 的第一版報告推上 Slack 之後，看了一眼說「有夠醜」。那個版本是列點式的，每一行格式不統一，訂閱名稱長度不一樣導致視覺歪歪的，看起來像是隨手把資料倒進去而已。

我當時不確定 Slack 能不能做出真正好看的表格。試了一下發現 Slack 的訊息不支援一般的表格語法，但支援等寬字體的程式碼區塊——也就是說，如果你能控制每一欄的字元寬度，就可以用 +---+---+ 這種格式做出有框線的視覺效果。

問題是中文字元佔兩格、英文字元佔一格，如果不處理這件事，對齊就會跑掉。我把這個問題描述給 Claude Code，它直接寫出了一個計算顯示寬度的函式，對每一個字元判斷是否為中日韓字元、對應佔兩格還是一格，然後用空白補齊每一欄到固定寬度。整個表格的對齊就這樣解決了。

同樣這個時機，我也發現原本技術部和行銷部混在同一張表的設計說不通——費用責任不一樣，當然要分開看。於是把一張表改成兩張，每張都有自己的小計。這個設計改動是在 n8n 的 Code 節點裡完成的，不需要動工具本身的程式碼。這讓我意識到，把組裝邏輯放在 n8n 而不是寫死在工具裡，調整起來靈活很多。

常見問題 FAQ

Q1：光貿電子發票 API 是什麼？需要申請什麼資格嗎？

光貿是台灣的電子發票加值中心之一，公司申請好電子發票服務之後，就能透過他們的 API 進行開立、作廢等操作。需要有公司的統一編號和向光貿申請的 App Key 才能使用。如果你的公司已經在用電子發票，問一下財務部門有沒有現成的 API 金鑰，說不定可以直接用。如果還沒有，也可以換成其他電子發票服務商——核心邏輯相同，只是 API 格式不一樣。

Q2：n8n 的 Code 節點要寫程式嗎？我不會寫 JavaScript 怎麼辦？

理論上要寫 JavaScript，但實際上你不需要自己從頭想邏輯。我的做法是：把需求說清楚，例如「我有一個訂閱清單，每筆有金額和幣別，我要把美金換算成台幣後加總」，然後交給 Claude Code 幫我寫節點裡的程式碼，我複製貼上進 n8n。你需要的不是寫程式的能力，而是把你要做的事說清楚的能力。這個差距比你想像的小很多。

Q3：我沒有工程師背景，這兩個工具做得到嗎？

做得到，但要有一個前提：你需要有一個可以部署工具的環境，例如 Zeabur、Railway 或類似的服務，讓你的工具有一個網址可以讓 n8n 去呼叫。這部分如果你完全沒接觸過，可能需要先花一點時間熟悉。工具本身的介面和邏輯，Claude Code 可以幫你做出來；n8n 的節點連接，照著流程學幾個小時就能上手。整體來說這個組合的門檻不是「要會寫程式」，而是「要會把需求說清楚、對工具有基本操作感」。

Q4：訂閱工具支援哪些幣別？換算匯率是即時的嗎？

目前支援台幣（TWD）、美金（USD）、歐元（EUR）、日圓（JPY）四種幣別。匯率是固定的估算值：1 USD ≈ 32 TWD、1 EUR ≈ 35 TWD、1 JPY ≈ 0.22 TWD。這不是即時匯率，但對於「大概花了多少」的估算來說已經夠用。如果你需要精確計算，可以在 n8n 裡加一個節點去抓即時匯率，這也是把邏輯放在 n8n 好調整的地方。

Q5：n8n 的每日提醒會不會每天都發一大堆訊息干擾頻道？

不會。三天到期提醒的邏輯是：每天早上跑一次，篩出剛好剩三天到期的訂閱，如果一項都沒有，流程就靜默結束，Slack 什麼都不會收到。只有在真的有東西要提醒的時候才會出現訊息。如果你覺得三天太晚、想改成五天，直接修改 Code 節點裡的那個數字就好，不需要動其他任何東西。

總結

這篇蓋了兩個工具——發票管理和訂閱監控——加上 n8n 串 Slack 的通知層。這兩個工具解決的問題很不一樣：發票是「我要記得做一件事」，訂閱是「我要記得一批持續發生的費用」；但兩個都有一個共同點：靠人記憶終究不可靠，自動化才是根本解法。

整個開發過程，我沒有自己寫任何一行程式碼。Claude Code 做介面和 API 串接，n8n 做排程和通知，我的工作是把每個工具該做什麼說清楚、把它們連在一起、遇到效果不對的地方說「這樣不行，改成這樣」。Vibe coding 的核心從來不是工具用得多熟，而是你能不能精準表達你要什麼。
