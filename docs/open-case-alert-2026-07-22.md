## 一、新增第四種合約：專案月收

不屬於 SEO，只提醒付款。**不限合約期間**，要停就自己下終止指令。

### 設計重點

跟其他三種合約不同，**專案月收不預先產生提醒日資料**。因為不限期間，沒辦法先算出 13 筆提醒列，所以改成「只存開案日，排程當天現算今天／明天是不是收款日」。好處是不用回頭補資料、可以無限期跑下去。

### 資料表（用節點 SQL 的 `CREATE TABLE IF NOT EXISTS` 自動建，沒去 Neon 手動開）

```
project_monthly_contracts          id / channel_id / channel_name / case_start_date / created_at
project_monthly_contracts_history  同上 + archived_at
```

### 新增節點

**建立線**（webhook path `project-monthly-charge`）

```
新增專案月收 → 專案月收開案日 → If3 ─true→ Slack4（日期格式錯誤提示）
                                    └false→ 封存舊專案月收合約 → 存入專案月收合約
                                            → 整理整理4 → 通知存入專案月收
```

**提醒線**

```
Schedule Trigger7（每天 10:30）→ 去資料庫找專案月收合約
  → 判斷要不要提醒5 → 模板文字和TAG人5 → 通知提醒5
```

### 提醒規則

- **開案當月不提醒**，從下個月起算
- 每月同一號的**收款日當天 ＋ 前一天**各提醒一次
- 該月沒有那一天（例如 31 號遇到 2 月）自動抓當月最後一天
- 文案 `📊 提醒: 請款開發票（專案月收）`，TAG Mike、Jena

例：開案 `2026-06-01` → 首次提醒 `2026-06-30`（前一天）＋ `2026-07-01`（當天），之後每月循環。

### 終止

`Code2` 加 `project` 類型。**「專案」的判斷必須排在「月收／月付」之前**，否則 `/終止專案月收` 會被「月收」關鍵字先抓走、誤判成月付合約。`封存終止合約` 的 SQL 加了 `oldp`／`archp` 段，並在最前面補 `CREATE TABLE IF NOT EXISTS`（表不存在會炸掉整個終止指令）。

## 二、業務規則修正：各類型合約可以並存

原本的規則「一個客戶頻道只會有一種收費方式」**是錯的**，實際上一個客戶可能同時有兩個專案進行。

因此當天稍早加的**跨表互清防呆全部拆除**——`封存舊合約`／`封存舊月收合約` 現在只封存自己那張表的同頻道舊約，不會互相刪。

## 三、「第 13 月」是編號取錯

SEO 和月收都一樣：`reminder_month = 13` 的日期就是 `contract_end_date`（滿 12 個月＝到期日），而編號 12 的那則落在**到期 d+2**。

以 SEO `2025-01-01` 開案為例：

| 內部編號 | 實際日期 | 意義 |
|---|---|---|
| 第 12 月 | 2026-01-03 | 到期後 2 天 |
| 第 13 月 | 2026-01-01 | 合約到期日 |

所以「到期 d+2 出最終成效報告」**不需要新增提醒日，日期本來就在**，只是名字取錯。

### 文字修正（日期算法完全沒動）

|---|---|---|
| SEO 第 12 月 | 第十二個月數據報表&請款開發票 | `📊 提醒: 最終成效報告 & 請款開發票` |
| 月收 第 12 月 | 第十二個月數據報表&請款開發票 | `📊 提醒: 最終成效報告 & 請款開發票` |

## 四、Bug 修正：29／30／31 號到期收不到「提前一個月」續約提醒

`判斷要不要提醒`（SEO）與 `判斷要不要提醒3`（月收）的 `month === 13` 原本判「**今天 +1 個月 == 到期日**」。遇到到期日是 29／30／31 號時，JS 的 `setMonth` 會溢位成下個月月初，永遠對不上，**提前一個月那次直接發不出來**，只剩到期當天一次。

實測：到期 `2026-03-31`、`2026-05-31` 都只觸發到期當天一次。

改成反向算「**到期日 −1 個月 == 今天**」，該月沒有那一天就抓當月最後一天：

| 到期日 | 提前一個月提醒日 |
|---|---|
| 2026-03-31 | 2026-02-28 |
| 2026-05-31 | 2026-04-30 |
| 2026-10-29 | 2026-09-29 |
| 2027-01-31 | 2026-12-31 |
| 2028-03-29 | 2028-02-29（閏年） |

到期日 1~28 號的行為完全不變。**只改判斷邏輯、沒動日期資料，現有合約立即生效不用重算。**

## 五、過期自動封存（新功能）

新增 `Schedule Trigger8`（每天 10:35）→ `自動封存過期合約` → `通知已封存`。

- **範圍**：SEO 年約 `contracts` ＋ 網站收費 `web_contract`（兩種都有明確到期日；要續約會額外開專案）
- **月收不做**（明確決定）；專案月收沒有到期日不適用
- **條件**：到期日 + 7 天 `<=` 今天（用 `<=` 不是 `=`，排程某天沒跑隔天會補做，不會漏）
- **不溯及既往**：SQL 寫死下限 `>= DATE '2026-07-22'`（功能上線日），更早就過期的舊約一律不動

封存 = 搬進對應的 `_history` 表（SEO 連 `reminders`／`progress` 一起清），不是刪除。封存後在客戶頻道發：

```
📦 <類型>合約已自動封存
客戶：X
到期日：Y（過期滿 7 天）
資料已移至歷史表，不再發提醒；要續約請重新開案
```

**未處理**：2026-07-22 當下已有 6 筆過期滿 7 天的 SEO 舊約不會被掃到（安好食 2/24、星羚貸款 3/17、12--謙心建築 4/10、魔力醫美 4/23、維諾娜珠寶 5/2、15--廣昱建設 6/10），要清得手動下 `/終止合約seo`。
